# 毕业设计（论文）管理系统需求获取与问题分析

## 1. 分析依据

本阶段的分析以 `docs/Requirement.md` 为主，以 `docs/Workflow.md` 和 `docs/Blueprint.md` 作为业务流程、时间安排与评分规则的补充依据。

本系统拟解决的问题包括：

1. 缺少开题评价、中期评价等过程控制环节
2. 开题与中期评价主体过于单一，需要增加评价教师
3. 评阅、答辩分组等操作复杂
4. 缺少宏观统计分析功能

## 2. 需求获取方案

### 2.1 需求获取目标

需求获取阶段的目标包括：

1. 明确毕业设计管理业务中的参与角色和职责边界
2. 识别课题管理、过程管理、评阅答辩、统计归档等核心流程
3. 找出现有系统的关键痛点与改进方向
4. 为后续用例模型、类图、顺序图和数据库设计提供事实依据

### 2.2 需求获取方法

本项目采用以下三类方式获取需求：

| 方法 | 目的 | 主要输出 |
|---|---|---|
| 文档分析 | 读取课程任务、业务流程和评分规则 | 需求边界、业务清单、时间约束 |
| 访谈法 | 了解不同角色在流程中的真实需求和问题 | 角色需求、问题清单、改进建议 |
| 场景分析法 | 以关键业务事件为主线梳理交互过程 | 用例、流程、异常情况 |

### 2.3 需求获取实施安排

| 阶段 | 内容 | 输出 |
|---|---|---|
| 第 1 步 | 阅读课程要求与业务材料 | 初步需求清单 |
| 第 2 步 | 确定访谈对象与问题 | 访谈提纲 |
| 第 3 步 | 梳理关键流程与异常场景 | 用例候选列表 |
| 第 4 步 | 提炼系统目标与范围 | 需求分析说明 |

## 3. 访谈对象设计

### 3.1 访谈对象

| 对象 | 关注重点 |
|---|---|
| 系统管理员 | 系统初始化、账号管理、截止日期设置、抽检、归档、统计 |
| 教学秘书 | 学院级流程管理、成绩发布、总评调整申请 |
| 教学院长 | 课题发布、学院统计、历史查询 |
| 专业负责人 | 课题审核、双选审核、任务书审批、评阅分配、答辩分组 |
| 指导教师 | 课题申报、任务书下达、开题审核、中期评价、论文审核 |
| 评阅教师 | 论文评阅、评阅意见与评分录入 |
| 答辩录入员 | 答辩分组、答辩成绩录入 |
| 学生 | 选题、材料提交、进度管理、论文提交、修改申请 |

### 3.2 访谈问题提纲

#### 对管理角色

1. 当前最耗时的业务环节是什么？
2. 课题审核、评阅分配、答辩分组过程中最容易出错的地方是什么？
3. 当前统计分析无法满足哪些管理需求？
4. 历史归档查询和导出最常见的使用场景是什么？

#### 对教师角色

1. 课题申报、任务书下达、过程材料审核是否存在重复操作？
2. 开题、中期、论文审核最需要哪些辅助信息？
3. 评阅和答辩相关流程中，哪些环节最容易遗漏？
4. 对修改申请和特殊情况处理最关注什么？

#### 对学生角色

1. 选题、材料提交、论文提交过程中最困惑的环节是什么？
2. 是否能清楚知道当前所处阶段与下一步任务？
3. 修改申请、退回修改、进度提醒是否足够清晰？
4. 对查询成绩、下载材料、查看流程状态有哪些需求？

## 4. 问题分析

### 4.1 当前系统主要痛点

| 编号 | 问题 | 影响 |
|---|---|---|
| P1 | 缺少开题评价、中期评价等过程性评价环节 | 过程管理薄弱，无法及时发现问题 |
| P2 | 开题、中期检查评价主体单一 | 评价结果客观性不足 |
| P3 | 评阅、答辩分组等操作复杂 | 管理成本高，容易出错 |
| P4 | 缺少宏观统计分析功能 | 管理层难以掌握全局进展和质量情况 |

### 4.2 直接改进目标

| 编号 | 改进目标 | 对应问题 |
|---|---|---|
| G1 | 增加开题评价、中期评价等过程控制节点 | P1 |
| G2 | 引入评价教师，形成多主体评价机制 | P2 |
| G3 | 优化评阅分配、答辩分组、成绩汇总流程 | P3 |
| G4 | 建立统计分析、导出、历史归档能力 | P4 |

## 5. 业务范围

### 5.1 系统范围内业务

1. 学院、专业、班级、账号和角色管理
2. 课题申报、审核、发布与选题
3. 任务书、开题报告、中期检查、论文等过程材料管理
4. 评阅教师分配、论文评阅、答辩分组、答辩成绩录入
5. 总评发布、成绩调整、评优推荐
6. 统计分析、表格导出、历史归档查询
7. 特殊情况处理，如课题调整、任务书修改、成绩退回等

### 5.2 系统范围外业务

1. 实际论文内容撰写质量评判细则的制定
2. 学校真实教务系统的集成接口开发
3. 真实数据库部署与正式生产环境建设
4. 非毕业设计相关的教务流程

## 6. 业务边界与关键约束

### 6.1 业务边界

- 本系统围绕“毕业设计（论文）全过程管理”展开。
- 核心对象不是单纯的学生信息，而是“课题、过程材料、评价记录、答辩组织、成绩与归档”。
- 系统必须体现多角色协作与审批链。

### 6.2 关键约束

1. 课题存在三种来源路径：教师盲选课题、教师指定学生课题、学生申报课题。
2. 任务书、开题、中期、论文、评阅、答辩、总评构成主流程。
3. 时间节点设置是系统的重要控制机制。
4. 总评成绩由指导教师、评阅教师、答辩小组三方形成，比例为 30% / 30% / 40%。
5. 系统必须支持过程信息统计、导出和历史归档。

## 7. 阶段性结论

通过需求获取与问题分析，可以得到以下结论：

1. 系统本质上是一个多角色、强流程、强审批、强状态的业务系统。
2. 建模重点应放在角色职责、材料生命周期、评价机制、统计归档和异常处理上。
3. 后续用例、类图和数据库设计必须显式体现“新增评价节点”和“多主体评价”。
